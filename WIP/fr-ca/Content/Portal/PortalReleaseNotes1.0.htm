Notes de la version 1.0 du portail

Le portail de GPSC est plus qu'une ferme de SharePoint. Elle est composé de multiples composantes qui travaillent ensemble, de ADFS, CRM, à Alphinat SmartGuide, aux applications Web personnalisées. Il est important que chacune de ces composantes soient comprises et planifiées avant que l'installation commence. Une installation du portail n'est pas complète à moins que toutes les composantes soient présentes.

La planification est essentielle pour déployer avec succès un portail. Tenter d'aller directement à la phase d'installation sans comprendre les composantes, où elles sont installées et de quelle façon elles communiquent, sera frustrant.

Exigences / Prérequis

  • Microsoft SharePoint 2016 sur place Le nombre de serveurs et la topographie de la ferme SharePoint relève du ministère.
    Consultez Installation de Sharepoint pour une discussion plus détaillée.
  • Microsoft SQL Server supporté par SharePoint 2016
  • Microsoft ADFS 2016 (requiert Windows Server 2016)
  • Optionnel : Microsoft Office Online Server, si le portail doit supporter la modification dans le navigateur des documents.
  • Alphinat SmartGuide, version actuelle. L'équipe du portail du GPSC a travaillé avec Alphinat pour intégrer les correctifs et les améliorations dans le SmartGuide pour appuyer les formulaires comme ils sont utilisés dans le portail.
  • Un serveur proxy inverse :
    • Si la ferme SharePoint a des multiples application frontale sur le Web (WEF), le proxy inverse doit supporter les « sessions sticky » ou les « sessions affinity ». Le proxy inverse doit être capable de diriger toutes les requêtes Web d'un client à la même WFE de SharePoint.
      SI la ferme SharePoint a seulement une seule WFE, cette exigence est levée.
  • Le proxy inverse doit être en mesure de diriger la circulation basée sur l'entête de l'hôte, ainsi que sur l'URL.
    P. ex. les requêtes pour https://<portal>/abc can be routed to a different back-end server than https://<portal>/123

    Toutes les mises en œuvre de proxy inverse qui respectent ces restrictions sont acceptables. La fonction de proxy inverse pourrait être mise en œuvre par un appareil, un pare-feu ou un un serveur Windows ou Linux. L'équipe du portail du GPSC a utilisé l’ensemble Application Request Routing gratuit de Microsoft pendant l'élaboration du portail de GPSC.

  • Le ministère doit être enregistré pour utiliser l'infrastructure de FJGC (CléGc et SecureKey). Le ministère doit avoir des certificats GJI pour leur portail et ils doivent avoir fourni un ensemble de métadonnées du  Portail valides au  FJGC. Cela doit être organisé par SPC.
  • Les noms DNS doivent être sélectionné et enregistré par le Portail, ADFS et le Domaine commun de la FJGC.
  • Les règles de pare-feu doivent être configurées pour passer la circulation entrante au serveur proxy inverse.
  • Les certificats SSL pour les tous les noms DNS du portail public (Portail, ADFS, Domaine commun de la FJGC)
    Consulter le document « Certificats SSL ».
  • Les certificats pour authentifier le service Web d'ADR et le service Web du portail. Ils peuvent être généré en utilisant OpenSSL, émis par un SC du ministère ou acheté comme certificats commerciaux.
  • Un plan de la topographie qui identifier les serveurs qui seront utilisé pour le portail et où ils seront situés. Les canaux de communication entre les serveurs doivent être compris et pris en compte.

Topographie

La topographie considère la façon dont les composantes du portail sont déployées sur les serveurs. Les règles d’acheminement de la circulation requises sur le serveur proxy inverse seront définies par ces choix.

La mise en œuvre de la référence est la même que le portail d'intégration du  GPSC a utilisé pendant l'élaboration. Cette mise en œuvre utilise des VM pour la plupart des composantes du portail.

Remarques :

  • Il est recommandé de garder le proxy inverse comme serveur séparé. Cela joute un couche supplémentaire d'isolation entre les serveurs du portail et Internet. Le proxy inverse termine et dirige toute la circulation provenant du Web.
  • Le serveur des ADSF de la FJGC peut probablement être combinée avec le serveur d'application pour héberger le SmartGuide d'Alphinat. Les  ADFS sont seulement accédés momentanément alors que les utilisateurs se connectent ou se déconnectent. La charge globale sur les  ADFS est extrêmement légère.
  • Le serveur des ADFS de la FJGC et le serveur des ADSF du GPSC doit être séparé l'un de l'autre. Il n'est pas possible de les combiner.
  • Le serveur d'application du  GPSC n'est pas nécessairement hébergé dans le même centre de données que les autres composantes du portail. Il devrait être hébergé logiquement « près » du CRM du  GPSC afin de minimiser le temps d'attente, alors que ces services Web doivent interagir étroitement avec le CRM.
  • Une production du déploiement de SharePoint devrait utiliser un serveur SQL séparé. N'installez pas le serveur SQL sur le même serveur que SharePoint.
  • Un déploiement extrême pourrait se retrouver avec toutes les composantes du portail empilées sur un serveur Cette configuration n'a pas été mise à l'essai et n'est pas recommandée
