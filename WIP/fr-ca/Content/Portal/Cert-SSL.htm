Certificat de SSL

Le portail nécessite un (ou plusieurs) certificats SSL délivrés par une autorité de certification de confiance publique, comme Entrust ou Verisign. Il est recommandé d'obtenir un certificat qui répond à tous les noms DNS requis par le portail. Toutefois, si on le préfère, il est possible d'utiliser un certificat SSL individuel pour chaque nom DNS.

Le portail doit avoir des certificats SSL qui répondent à ce qui suit :

  • Le nom DNS du portail
  • Le nom  DNS utilisé par les des services ADFS (Active Directory Federation Services) (s'il est différent du nom DNS du portal)
  • Le nom DNS émis pour le portail dans le domaine commun de la FJGC (fédération des justificatifs du gouvernement du Canada) (le domaine fjgc-gccf.gc.ca). Ce nom DNS est émis par le groupe de gestion des justificatifs externes (GJE) de Services partagés Canada  (SPC) dans le contexte de l'intégration d'un nouveau portail.

Chaque nom est ajouté au certificat SSL en tant qu'autre nom de l'objet, ou SAN. Il est aussi possible d'utiliser de SAN avec caractères génériques, qui permettront de répondre pour de multiples domaines.

Les certificats SSL sont acquis par l'entremise de SPC.

Installation

Le certificat SSL du portail doit être installé sur le serveur du proxy inverse, étant donné que le proxy inverse sera le point de terminaison de tout le trafic entrant d'Internet. Il doit aussi être installé et configuré dans ADFS. Si les connexions entre le proxy inverse et les autres serveurs sont en SSL, le certificat SSL peut aussi être utilisé pour SharePoint, SmartGuide, l'application  Web des extensions ADFS et l'application Web de domaine commun de la FJGC.

Exemple – APECA en direct

Pointez votre fureteur sur https://direct.acoa-apeca.gc.ca et visionnez les propriétés du certificat de l'APECA. L'APECA utilise un seul certificat, qui est valide pour tous les serveurs Internet qu'elle choisit d'héberger, de même que le domaine commun de la FJGC pour les environnements de développement, d'essai et de production.
